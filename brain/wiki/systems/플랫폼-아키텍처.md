단일 Next.js 15 앱이 host 기반 미들웨어 라우팅으로 dkansim.com/hq/report/agent/contents
5개 서브도메인을 서빙한다. 저장소는 로컬 JSON과 Supabase(Postgres+Storage) 사이를 env로
전환하는 이중 모드다. Vercel에 배포되며 GitHub Actions가 CI/자동배포/자가개선 파이프라인을 맡는다.

## 핵심 사실

- 서브도메인 라우팅: `dkansim.com`(메인), `hq.dkansim.com`(`/hq`), `report.dkansim.com`(`/report`),
  `agent.dkansim.com`(`/agent`), `contents.dkansim.com`(`/contents`) — 모두 host 기반 미들웨어 rewrite.
- 저장소 이중 모드: `SUPABASE_ENABLED`(`src/lib/supabase-server.ts`), `isSupabaseReservationsDbReady()`
  (`src/lib/supabase-pg.ts`) 로컬 dev는 `data/*.json`, 프로덕션은 Supabase Postgres/Storage.
- 인증 3계통, 전부 서버 전용 쿠키: admin(`dk_admin_auth`), resident(`dk_resident_auth`, HMAC),
  worker(`dk_worker_auth`, JWT-like) — `src/lib/admin-auth.ts`, `resident-db.ts`, `worker-session-server.ts`.
- 마이그레이션은 `supabase/migrations/`에 001부터 순번으로 쌓임, `npm run db:apply`로 적용.
- 배포는 `git push`만으로 자동 안 됨 — `npx vercel --prod` 수동 실행 필요.
- **⚠️ Vercel 프로덕션은 로컬 파일시스템에 쓴 내용이 지속되지 않는다**(배포 이미지가 읽기
  전용). `fs.writeFile()`로 뭔가를 영구 저장하려는 코드는 로컬 dev에서는 멀쩡히 동작해서
  이 문제가 안 보인다 — 실제 사례로 백업 시스템이 이걸 몰라서 프로덕션에서 몇 달간
  아무것도 저장 안 하고 있었음(→ [[2026-확정-기술결정]] "백업/복구 시스템 전면 수정" 참고).
  영구 저장이 필요하면 Supabase Storage/Postgres를 써야 한다.
