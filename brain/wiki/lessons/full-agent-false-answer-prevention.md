Built the "거짓 답변 방지" system on top of [[full_agent_review_criteria_fix]] and [[trusted_domains_and_document_gen_status]]. Key discovery during STEP 1 investigation: `chatWithFullAgent` (the actual "풀 에이전트" / 총괄 path in `full-agent.ts`, used for every non-attachment message) had **no RAG search at all** — only the other 8 named agents (`chatWithAgentPlus` in `agent-chat.ts`) called `searchKnowledgeChunks`. Had to build that in first.

**What was added**:
- `src/lib/knowledge-chunks-search.ts`: `searchKnowledgeChunksWithEvidence()` — same `match_chunks` RPC as the existing `searchKnowledgeChunks`, but returns structured `{chunks, hasEvidence, evidenceSummary, contextBlock}` filtered to similarity ≥ 0.7, instead of a flat prompt-ready string.
- `full-agent.ts` `chatWithFullAgent`: runs this RAG search every turn (parallel with history/snapshot/routing), injects the evidence block (or a "no evidence found, don't overclaim" notice) into the user-turn message content (not the cached system prompt — kept separate to not break prompt caching). Added a "정직성 규칙" (honesty rules) block to `FULL_AGENT_SYSTEM_PROMPT`: don't assert unverified facts confidently, don't claim unimplemented features exist, don't misstate cost/time, say "확인이 필요합니다" when unsure. **Also fixed a real bug while doing this**: `chatWithFullAgent` used to persist the assistant reply to `agent_chat_messages` *before* route.ts's Gemini review could block/correct it — meaning a blocked answer would still show up in chat history on reload. Moved that persistence to `route.ts` (`appendChatMessage("general","assistant", reviewed.reply)`), after review completes, so history matches what's actually shown.
- `cross-validate.ts` `validateAgentAnswer`: added `hasRAGEvidence` param and `hasFalseInfo` return field (alongside the existing `hasDangerousMisinfo` from [[full_agent_review_criteria_fix]]). Criteria: 허위/과장정보, 미구현기능 오안내, 위험한 오정보, 비용/시간 왜곡, 욕설/비하. **Critical, empirically-tested wording**: explicitly tells Gemini that "no RAG evidence" and "confident general-knowledge answers" are NOT defects by themselves — this is the same class of regression as [[full_agent_review_criteria_fix]] and was verified with 4 direct Gemini API test cases *before* deploying (general off-topic question → 100/pass; fabricated "already-implemented" feature claim → NOT caught, known limitation below; genuine dangerous electrical-safety misinfo → correctly caught with score 0 + correction; RAG-backed correct answer → 100/pass).
- `route.ts` `reviewFullAgentAnswer`: blocks with a fixed "문의해 주세요" message when `hasFalseInfo || hasDangerousMisinfo`; substitutes Gemini's correction when it fails but has one; shows a `no_evidence` badge (not a block) when there's just no RAG match. `logResult`'s meta now includes `hasFalseInfo`/`hasDangerousMisinfo`/`hasRAGEvidence` as structured JSON fields (previously only embedded in the free-text verdict — fixed in commit d0b638c after noticing the user's own completion criteria required querying it).
- `chat-client.tsx`: 4-state badge (✅ ok / ⚠️ no_evidence / ⚠️ corrected / 🚫 blocked).

**Known limitation (found via testing, not fixed — inherent)**: Gemini has no manifest of what's actually implemented in this specific codebase, so it cannot reliably catch claims like "카카오페이 자동결제가 이미 구현되어 있습니다" when that's false — it only catches fabrication when Gemini's *own* general/domain knowledge is strong enough to recognize the claim as wrong (this works great for electrical-safety facts, not for company-specific feature claims). Building a reliable feature-manifest cross-check would be a separate, much larger project — flagged to the user rather than silently overclaiming coverage.

**Production verification** (2026-07-02): live chat questions confirmed (a) off-topic general question scores 100/no false positive, badge=no_evidence as expected; (b) a leading/loaded dangerous-misinfo *question* got a factually-correct answer from Claude itself (didn't need the Gemini backstop — good defense-in-depth signal); (c) a real legal-citation mixup ("전기사업법 제22조" should've been "전기안전관리법 제22조") was caught by Gemini as a warning (score 95, not blocked — correctly calibrated, a citation-number slip isn't "위험한 오정보"); (d) `agent_logs` meta confirmed to contain structured `hasFalseInfo`/`hasDangerousMisinfo`/`hasRAGEvidence` fields.
