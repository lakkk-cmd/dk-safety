`video_jobs`가 `pending_review`에 도달하면 대장이 승인/반려하고, **승인만으로는 발행되지
않는다** — 로컬 PC 워커가 `approved` 상태를 폴링해서 실제 유튜브 업로드까지 수행한다.

## 절차

1. hq 승인 대시보드에서 `pending_review` 상태 작업의 렌더링된 영상(`video_path`)과 검증 점수
   (`validation.score`)를 확인한다.
2. 문제 없으면 승인 → `approveVideoJob(id)`(`src/lib/video-jobs.ts`)가 `pending_review`인
   행만 `approved`로 전환. **이 시점엔 아직 유튜브에 안 올라감.**
3. 로컬 PC 워커(`worker/index.mjs`)가 `approved` 상태를 폴링해서 `uploading`으로 선점 →
   유튜브 비공개 업로드 → 성공 시 `published`(+`youtube_url` 기록), 실패 시 `error`.
   공개 전환은 유튜브 스튜디오에서 별도로 해야 함(코드가 자동 공개하지 않음).
4. 문제 있으면 반려 → `rejectVideoJob(id, note)`가 `rejected`로 전환.
5. `published`/`error`/반려 시 자동으로 `brain/raw/external/`에 결과가 기록된다
   (워커의 published/error는 `worker/lib/append-to-raw.mjs`로 로컬 파일쓰기, 반려는
   Vercel 쪽 `appendToRaw`로 GitHub API 커밋 — → [[영상-팩토리]]).

## 확인할 것

- 승인 게이트는 워커 코드 자체에 강제돼 있음 — `approved`가 아닌 행은 업로드 함수가 절대
  집어가지 않는다(`supabase/migrations/063_video_jobs.sql:5` 주석).
- 로컬 워커가 꺼져 있으면 `approved` 상태에서 멈춰 있을 수 있음 — 워커 실행 여부부터 확인.
